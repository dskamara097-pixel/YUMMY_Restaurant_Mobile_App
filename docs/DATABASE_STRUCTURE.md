# Database Structure

Project: YUMMY Restaurant Mobile Application  
Authority: `PROJECT_BIBLE.md`

## Overview

YUMMY uses Cloud Firestore as the primary database. Screens do not call Firestore directly. Data access flows through hooks and repository classes:

`Screen -> Hook -> Repository -> Firebase service -> Firestore`

Document IDs are generated by Firestore unless a stable ID is required, such as active customer cart documents.

## Collections

### users

Purpose: stores customer, vendor, admin, and rider profile records.

Main fields:

- `id`
- `uid`
- `fullName`
- `username`
- `usernameLower`
- `email`
- `phone`
- `role`: `customer`, `vendor`, `admin`, `rider`
- `status`: `active`, `pending`, `disabled`
- `photoURL`
- `createdAt`
- `updatedAt`

Access:

- Users read/update their own profile.
- Admins manage all users.

### restaurants

Purpose: stores restaurant/vendor business profiles.

Main fields:

- `id`
- `ownerId`
- `name`
- `description`
- `categoryIds`
- `logoUrl`
- `coverImageUrl`
- `rating`
- `reviewsCount`
- `deliveryTimeMinutes`
- `deliveryFee`
- `active`
- `status`: `pending`, `approved`, `rejected`, `suspended`
- `createdAt`
- `updatedAt`

Access:

- Public can read approved/active restaurants.
- Vendors manage owned restaurants.
- Admins manage all restaurants.

- `name`
- `description`
- `active`
- `sortOrder`
- `createdAt`
- `updatedAt`

Main fields:

- `id`
- `restaurantId`
- `categoryId`
- `name`
- `description`

- `currency`
- `imageUrl`

- `available`
- `featured`
- `createdAt`

Access:

- Public can read available foods.

- Admins can manage all foods.

### carts

Purpose: stores active customer cart data.

Main fields:

- `id`
- `customerId`
- `restaurantId`
- `items[]`
- `subtotal`
- `updatedAt`

Cart item fields:

- `foodId`
- `restaurantId`
- `restaurantName`
- `name`
- `unitPrice`
- `quantity`
- `lineTotal`
- `imageUrl`

### orders

Purpose: stores placed orders.

Main fields:

- `id`
- `customerId`
- `restaurantId`
- `items[]`
- `subtotal`
- `deliveryFee`
- `serviceFee`
- `discount`
- `total`
- `status`: `pending`, `paymentReceived`, `preparing`, `ready`, `delivered`, `cancelled`

- `paymentId`
- `riderId`
- `deliveryAddressId`
- `notes`
- `createdAt`
- `updatedAt`

### payments

Purpose: stores dummy assignment payment records.

Main fields:

- `id`
- `orderId`
- `userId`
- `restaurantId`
- `method`: `cashOnDelivery`, `dummyMobileMoney`, `dummyCard`
- `status`: `pending`, `paid`, `failed`, `refunded`
- `amount`
- `currency`
- `transactionReference`
- `failureReason`
- `createdAt`
- `updatedAt`

Important: no real card numbers, bank credentials, or mobile money secrets are stored.

## Storage Paths

Firebase Storage helper paths:

- `profile-images/{userId}/{fileName}`
- `restaurant-images/{restaurantId}/{fileName}`
- `food-images/{restaurantId}/{fileName}`
- `review-images/{userId}/{fileName}`

## Security Summary

- Customers manage their own profiles, carts, orders, payments, notifications, addresses, and favorites.
- Vendors manage owned restaurant data, menu items, categories, coupons, offers, and restaurant orders.
- Admins manage all collections.
- Riders access only assigned orders.
- Public reads are limited to approved restaurants and available foods.
